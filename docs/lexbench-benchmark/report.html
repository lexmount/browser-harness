LexBench-Browser · Domain-Skill Ablation

技能库让弱模型少走弯路——而且换个任务照样管用

给 50 个站点各写一份 domain skill，让 Haiku 在云浏览器上执行数据集任务。对照"有技能 / 无技能"两臂，并在技能被调过的任务与另一个没见过的任务上分别测量。技能的收益几乎全部迁移到了新任务上。

123%

泛化是这次实验最关键的发现。同一份 skill（在 case A 上调过），用到完全不同的 case B 上，增益不降反升（+13pp → +16pp）。说明 skill 沉淀的是可迁移的站点知识——接口地址、DOM 选择器、反爬绕法、字段不可得的诚实标注——而不是某个具体任务的答案。写一个 case 的 skill，整站的同类任务都受益。

按本 case 的成功增益排序，被拖累的站置顶标红。步数为代理自报的 browser-harness 调用次数（弱模型有低报，趋势可比）；token 为轨迹的全量消耗合计：输入 + 缓存写入 + 缓存读取 + 输出，按 API 调用去重后累加（真实、不可伪造）。个别站步数降但 token 升（如 pptv：技能改走浏览器渲染，页面大），是路径取舍差异。 任一臂为 infra 的格子，步数/token 已置灰、不参与对比（那是会话崩溃前的残值）。

方法与诚实边界

  • 执行体 Haiku 4.5，云浏览器为 Lexmount（原生 CDP 桥接 browser-harness），两臂唯一变量是能否读 skill 文件。
  • 成功率由独立 Fable 判分员对数据集 reference_answer 的 key_points 逐站判定；动态数据（股价/汇率/最新文章）只要结构正确、像真实抓取即算通过。
  • 安全剔除：google.com（14 个任务中 13 个是人肉/点击欺诈/炸弹教程等拒绝测试）与 taobao 整站排除，不为恶意任务建或测 skill。
  • 基础设施噪声：早期云会话上限（50 持久上下文）导致部分站 infra 失败，已定位为"每任务新建 context 且只关 session 不删 context"的编排 bug——改用无状态临时会话后消除，配对分析已剔除受影响站。
  • 步数口径：部分弱模型代理绕过计量 wrapper 改用其它浏览器工具，故步数以 harness 的工具调用总数为准，跨代理趋势可比、逐站不严格。
